# Live Agent Configuration

> Document status: CURRENT SUPPORTING GUIDE.
> Current authority: `scripts/factoryctl.py`, schemas, tests and current public guides.
> Runtime boundary: This guide explains live profile setup; executable gates and current validation still decide whether work can move.

The factory has runtime layers plus a capability-pack gate. Keeping them
separate prevents weak autonomous execution.

## Layers

| Layer | File | What it means |
|---|---|---|
| Worker | `agents/worker-registry.public.json` | The process role: when this worker is required and which receipt field it must produce. |
| Agent profile | `agents/worker-profiles.public.json` | The agent identity: mission, operating rules, authority, refusal conditions, evidence, review, handoff and learning contract. |
| Hermes binding | `agents/hermes-profile-bindings.public.json` | The runtime route: Hermes profile name, queue policy, skill refs, result schema, manifest refs and evidence path policy. |
| Profile readiness ledger | `agents/worker-profile-readiness.public.json` | The public-safe ledger that says whether a profile has current smoke/eval readiness or only a degraded contract record. |
| Worker packet | generated by `scripts/factoryctl.py` | The card-specific task given to the agent, including input contract, output contract and profile binding. |
| Capability pack | `agents/capability-packs.public.json` | The product-type coverage gate: whether the Factory has ready specialists for the surfaces in a card. |

## Operator Types

The public roster uses the word worker for every routable operator, but the
runtime treats them differently:

| Type | Meaning | Example |
|---|---|---|
| Planner/router | Converts source, phase, risk and scope into the next packet. | `source-ledger-worker`, `product-architect`, `decomposition-planner` |
| Builder | Creates implementation artifacts for a specific product surface. | `frontend-builder`, `backend-api-builder`, `solana-quasar-builder` |
| Reviewer/proof runner | Verifies, challenges or packages work done by another operator. | `independent-reviewer`, `autoreview-gate`, `remote-proof-runner` |
| Specialist gate | Blocks a risk surface until evidence exists. | `codex-security`, `appsec-owasp-specialist`, `supply-chain-gate` |
| Human-support gate | Records a real human decision. | `human-gate-clerk` |
| Interface profile | Talks to the human and mirrors cockpit state without executing product work. | `overkill-factory-gerente` |

This distinction matters. A gate can be operable without being a builder, and a
builder can be autonomous without being allowed to approve itself.

## Conceptual Role Drift Rule

Some canonical-stage names are intentionally plain: Method Router, Product
Outcome Worker, Product Experience Router, Security Architect, Production
Readiness and similar names. In runtime they are mapped to registered workers in
`docs/agents/factory-stage-agent-map.md`.

Do not create separate Hermes executor profiles for those conceptual names by
default. A loose profile with a good-sounding name is weaker than an official
worker because it has no registry entry, no binding, no permission class, no
packet route and no validation proof. If such a profile appears in Hermes, treat
it as profile drift unless it has been promoted through the worker contract.

The dedicated exception is `overkill-factory-gerente`: it is a gateway/interface
profile for Discord. It may talk to the operator, receive inputs and register
intent, but it must not choose specialists outside the Kanban, execute material
work or mark cards done.

## Builder Layer

The factory has surface-specific builders so product execution is not handled
by one vague developer profile:

- `frontend-builder` for screens, components, responsive states and wallet UI.
- `backend-api-builder` for APIs, auth/session, services and validation.
- `data-persistence-builder` for schema, migration, storage and data access.
- `solana-quasar-builder` for Solana program code using Quasar.
- `solana-quasar-qa-engineer` for Quasar/devnet/local onchain behavior proof.
- `wallet-transaction-builder` for wallet connection, signing prompts and
  transaction states.
- `integration-builder` for cross-surface end-to-end wiring.
- `test-automation-builder` for tests, evals and regression proof.
- `infra-devops-builder` for CI/CD, runtime, deploy and rollback wiring.
- `agent-runtime-builder` for Hermes/factory adapters, profiles, skills and
  worker routing.

`implementation-worker` is fallback only. It is required only when a card is
implementation/code work and no specialist builder surface matches.

## Capability Pack Gate

Builder routing is not enough for modular products. Before execution, the
Factory also checks whether the product type is covered by a ready capability
pack.

Ready core packs can continue to worker routing. Template packs block until a
`capability_pack_contract` activates the missing specialist set. For example,
ordinary web/API/data work can use `web-saas-core`, but native mobile, desktop,
game, AI/ML, fintech, regulated, analytics, browser extension and hardware
products require pack activation before material execution.

This prevents a generic implementation worker from silently taking work that
needs a real domain specialist.

## Hermes Alignment

Hermes profiles are separate agent homes. A profile carries its own config,
credentials, sessions, skills, memory, state and `SOUL.md`. The factory does
not rely on a profile name alone:

- profile description helps Kanban routing;
- `SOUL.md` defines personality and standing instructions;
- skills define procedures and load only when needed;
- config/toolsets define what the profile can access;
- `terminal.cwd` defines where tool execution starts;
- a smoke run must prove the profile can execute the expected worker packet.

This is why every factory worker now needs both a profile and a binding. A
worker without a binding is a role on paper, not an executable agent. The
factory fails closed when the binding manifest is missing or incomplete, and
worker packets require `profile_binding`.

## Runtime Manifest

Every binding points to `docs/agents/hermes-profile-manifest.md` and a
public-safe smoke record. The manifest requires:

- real profile name;
- non-bare profile description;
- `SOUL.md`;
- expected skill refs;
- least-privilege tool policy;
- no direct card-state mutation;
- smoke evidence.

This closes the gap between "agent described in JSON" and "agent exists in the
Hermes runtime".

`last_hermes_smoke_ref` is not enough by itself. The public readiness ledger
must also state the worker id, profile id, packet fixture, smoke result, eval
result, timestamp, producer and freshness policy. The checked-in public ledger
is intentionally degraded: it proves that the contract is mapped, not that the
runtime profile is currently ready.

Only a fresh sanitized ledger row with `readiness_state=current_profile_ready`,
`smoke_result=PASS`, `eval_result=PASS` and a non-stale freshness policy may be
treated as current profile readiness. Stale, missing or wrong-profile rows stay
blocked or degraded.

## Queue Rule

The binding carries `dispatch_queue_policy`, but it is not a second runtime
queue source. The executable queue is computed by
`factoryctl.worker_queue_class` and exposed as `worker_task.queue_class`.

That is better for Hermes because phase/risk can change the queue. A static
binding queue would let the adapter and transition plan disagree.

## Result Schema Rule

The result schema promised by the binding must match the worker result emitted
by `factoryctl`:

- Product Face emits `schemas/product-face-result.schema.json`;
- Solana/Quasar Auditor emits `schemas/auditor-result.schema.json`;
- human gate emits `schemas/human-gate-record.schema.json`;
- builders and generic workers emit `schemas/worker-result.schema.json`.

If the promised schema and emitted schema differ, the result is invalid.

For a card to reach `done`, every required builder must produce its receipt
field. For example, a Solana/Quasar implementation card can require both
`solana_quasar_build_result` and `solana_quasar_qa_result` before the normal
Auditor, security, QA, review and human gates can close.

## Profile Contract

Each profile must define:

- identity and mission;
- operating rules;
- activation phases, surfaces and trigger words;
- authority: what it may do, must not do and when human gate is required;
- tool policy;
- required inputs and refusal conditions;
- output receipt field and required sections;
- evidence trust floor and invalid evidence examples;
- review boundary;
- handoff contents;
- operator-understanding record;
- bounded failure policy;
- promotion and demotion criteria.

Security-critical profiles must also define:

- domain contract;
- waiver contract;
- routing moments;
- required controls;
- minimum evidence;
- sensitive surface triggers.

The operator-understanding record matters because high-quality agents should
not only finish work; they should leave the human able to explain what was done,
why it was done, what can break and what remains open.

## Why This Is Better

A long prompt can sound intelligent while still being impossible to operate. A
profile contract is better because it can be validated, bound to a Hermes
profile, included in worker packets and reviewed independently.

The factory uses:

- skills for reusable procedures;
- profiles for identity and authority;
- packets for task-specific input/output;
- results for proof;
- gates for refusing weak transitions.

That split is heavier than a single prompt, but it is better for autonomous
agents because every boundary is inspectable.

## Review Loop

Agent profiles are not considered final just because they are written. They
must pass:

```bash
python scripts/validate_worker_profiles.py
python scripts/validate_public_json_artifacts.py
python -m unittest tests.test_worker_profiles -q
python -m unittest tests.test_factoryctl -q
```

Then an independent reviewer should score each profile on:

1. identity clarity;
2. activation precision;
3. authority and refusal strength;
4. evidence specificity;
5. Hermes operability;
6. failure and handoff behavior;
7. public-safety hygiene.

Any score below `10/10` creates a fix packet and another review round.
