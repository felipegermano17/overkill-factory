# Factory 11 Critical Review

## What Is Still Weak

1. Worker routing is broader, but many workers still produce packets and
   contracts rather than executing real tools automatically.
2. Security is now a matrix, but not every matrix row has a dedicated automated
   fixture.
3. Product Face proof is defined, but the browser/a11y/mobile runner still needs
   full automation.
4. Hermes update safety exists as manifest and checks, but no disposable Hermes
   update smoke was run in this pass.
5. Remote proof is defined, but no Crabbox/Testbox/container run was executed in
   this pass.
6. Context spine is defined as a safe model, but no memory storage backend has
   been tested.

## What Can Break In Practice

1. Agents may treat `requires_execution` as a blocker forever instead of
   spawning the right worker.
2. Workers may produce structured JSON with weak evidence.
3. High-risk tasks may become slow if every specialist runs on every card.
4. Dashboard/API paths in Hermes may bypass CLI-tested gates unless explicitly
   validated.
5. Public safety scan may miss a new class of private leak unless the denylist
   evolves.
6. The factory may overfit to the dry pilot until multiple real contexts are
   tested.

## What Agents Could Misread

1. AutoReview is evidence, not approval.
2. Handoff is replayable state, not a summary.
3. Auditor preflight is not real program audit.
4. Product Face packet is not rendered proof.
5. Memory is not truth.
6. A green local preflight is not production readiness.
7. A waiver is not a shortcut; it needs owner, evidence and expiry.

## What Needs To Be More Objective

1. Worker result schemas for all new workers.
2. Reconciler: required workers vs delivered worker results vs closure summary.
3. Product Face visual overlap and a11y thresholds.
4. Security control pass/fail criteria per domain.
5. Remote proof TTL, cost and cleanup fields.
6. Closed-specialist promotion threshold tracking.

## What Is Too Sophisticated

1. The full worker roster is too much for small R0/R1 tasks.
2. The security matrix can become heavy if blindly applied to every card.
3. Remote proof is powerful but can add latency and cost when local proof is
   enough.

Mitigation: keep gates risk-tiered and worker-triggered, not universal.

## What Is Too Simple

1. Hermes adapter compatibility check currently verifies markers, not a live
   disposable runtime.
2. Public safety scan is denylist-based, not semantic.
3. CI is minimal: unit tests, compatibility marker check and public scan only.
4. No real Product Face runner exists yet.
5. No full Codex Security scan report exists yet for the public repo.

## Score

Factory 11 is strong as an operational hardening pass: 9.5/10 for methodology
and public-safe agent operability.

It is not a final 10/10 because the remaining proof requires heavy validation
across multiple real contexts, including a real product paper, full security
scan, Hermes update smoke, Product Face automation, remote proof and real
Solana/Quasar implementation audit.
