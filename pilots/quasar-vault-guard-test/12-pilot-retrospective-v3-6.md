# F18 Pilot Retrospective For V3.6

## What Was Too Heavy

- Requiring a full Codex Security exhaustive scan for a tiny dry-pilot artifact
  would be too heavy. The method needs a scoped scan tier.
- Whimsical MCP not being exposed as native tools forced direct JSON-RPC use.
  The method needs a documented fallback.

## What Was Too Vague

- "Automate worker" was ambiguous between creating worker packets, running the
  worker, and collecting final evidence.
- "Human approval" needed a clearer distinction between test authorization and
  production approval.
- Hermes V3.5 and Hermes V2 gates are both active. The factory must validate
  the compound contract locally before attempting completion.

## What Agents Could Misread

- Auditor preflight as Auditor pass.
- Product Face Packet as Product Face evidence.
- Worker packet as worker result.
- Gate report as Hermes completion.
- A preflight gate report with `requires_execution` as a post-execution failure.
- A dry-pilot approval as production/deploy authority.

## What Should Become Stricter

- Receipt Five should explicitly require all required worker result fields for
  cards whose gate report has `requires_execution`.
- Hermes should store `gate_report_ref` and `worker_packet_refs` on the card.
- The map should show "packet is not evidence" as a first-class warning.
- `security_scan_result` must use the Hermes completion shape:
  `scanner_agent`, `tool`, `scope`, `result`, `findings_summary`,
  `evidence_refs`.
- High-risk Hermes receipts need V2 metadata too: evidence paths,
  verification, sandbox invariants, rollback evidence and scoped approval
  records.

## What Should Become Simpler

- The first pilot folder shape should be generated by a command.
- Worker result creation should support assembling Receipt Five automatically.
- Whimsical board operation now has a repo script wrapper for the direct MCP
  path.
- The context-lock package should be portable. It needs its own validator copy or
  a mounted repo path, otherwise Hermes reviewers cannot re-run checks.

## Hermes Review Findings

Hermes `qa-reviewer`, `security-reviewer`, `security-runner`, `architecture-reviewer` and
`independent-reviewer` all passed the dry pilot with hard boundaries.

Shared findings:

- The pilot is valid as a process proof, not as product proof.
- No real Quasar program exists, so no onchain safety is proven.
- Preflight gate reports must not be read as post-execution status.
- The package must be independently re-runnable.

## V3.6 Decision

Promote the Factory 10+ distinction:

```text
preflight gate report -> worker packet -> worker execution
-> worker result -> closure summary -> Receipt Five
```

This is better than the previous model because it prevents agents from treating
handoff as evidence.

Add the Hermes compatibility rule:

```text
local factory validator must fail before Hermes fails
```

This is better than discovering missing fields at Kanban completion time, because
agents can correct the artifact before creating or promoting a durable card.
